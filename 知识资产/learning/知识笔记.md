## 主题：当业务约束已收敛为“单任务单凭证”时，执行阶段必须只认任务快照
**一句话结论**：如果业务已经明确一个执行任务只能使用一个 `credentialId`，那么配置层只负责选择来源，任务快照才是执行真相；发送时继续回读模板配置，会让“快照”失去意义。

**解决什么问题**：这能避免配置后改、模板后改或多渠道兼容逻辑把已经生成的任务再次漂移，导致重试、补发和审计无法复现当时实际使用的凭证。

**核心原理**：编辑态和执行态是两套不同语义。编辑态关注“运营如何配置”，适合把凭证放在配置表；执行态关注“这次任务到底用了什么”，适合把凭证冻结到任务表。只要业务前提是“一个执行任务只使用一个凭证”，执行器就应该只读取任务快照，而不是重新回到模板配置层做二次选择。

**常见设计取舍**：把凭证只放在配置层，改动最少，但历史任务会受后续配置修改影响；把凭证同时固化到任务层，能保证执行可追溯，但前提是执行阶段真正只认任务表。若任务内部仍存在语言或模板级多凭证差异，就不能偷懒用单一任务快照表达。

**排障经验**：遇到问题：表里和代码里都出现了 `credentialId`，但发送阶段仍然没有按快照执行。原因：只补了字段，没有同步收口读取路径。解决方案：把检查固定成四层审计：表结构有没有列、实体类有没有字段、任务生成时有没有写入、发送时有没有只从任务表读取。

**可复用经验**：任何“配置选择 + 任务执行”类型的系统，只要要做重试、审计、补发或冻结历史状态，都应该先明确“谁负责选择、谁负责冻结、谁是执行唯一读取源”。

**延伸思考**：如果未来重新引入语言或模板级多凭证差异，需要把“单任务单凭证”的约束显式升级为更细粒度的快照模型，而不是继续在单任务表上堆兼容字段。

## 主题：迁移方案落地时，最容易漏的是“实体类与真实表结构已经脱节”
**一句话结论**：迁移失败不一定是业务方案错了，很多时候是库表、实体、复制逻辑和执行读取路径只改了一半，形成了“表里有列、实体没字段”或“实体有字段、库里没列”的半对齐状态。

**解决什么问题**：这能减少“代码看起来已经支持新方案，但真实运行仍然读旧字段、写空值或根本没落库”的隐性问题。

**核心原理**：ORM 和 Bean 复制链路默认相信模型定义。如果真实表结构和实体类不一致，问题会在持久化或运行时被静默吞掉，而不是立刻爆成显眼错误。迁移时必须同时审查数据库 schema、PO/VO、任务生成链路、执行读取链路，四层缺一不可。

**常见设计取舍**：只做字段重命名或只补 SQL 脚本，推进速度快，但非常容易留下半迁移状态；把库表、实体、生成器、执行器一次性对齐，工作量更大，却能显著降低后续排障成本。

**排障经验**：遇到问题：某些渠道的任务表已经有 `credential_id`，但 Java 实体没有；另一些模板实体声明了 `credentialId`，真实表却没有对应列。原因：历史改造只覆盖了部分仓库或部分层次。解决方案：把“字段存在性检查”从命名层升级为“schema -> entity -> snapshot write -> runtime read”的闭环核对。

**可复用经验**：跨渠道、跨仓库迁移时，不要只搜字段名；要把每个渠道放进同一张核对表里，逐项确认配置实体、任务实体、模板实体、VO、生成逻辑和执行逻辑是否都与方案一致。

**延伸思考**：可以继续沉淀一套“迁移方案落地审计清单”，把字段、表结构、运行时读取源和回归验证点固化成固定模板。

## 主题：RCS webhook 里的消息方向、计费分类和会话归并是三套不同概念
**一句话结论**：`in / out` 只能回答消息方向，`Basic / Single / Conversation` 才是计费分类；如果供应商 webhook 不提供明确 `conversationId`，会话归并和日账单只能由本地规则建模后再与供应商数据对账。

**解决什么问题**：这能避免把方向字段误当成计费类型，也能避免因为 webhook 里没有显式“会话转换标识”，就错误地把每条消息直接映射成账单结果。

**核心原理**：方向是事件属性，计费是业务归类，二者是正交维度。CM.com 文档里可以用 `reference` 和 RCS `Incoming Events` 的 `messageContext` 把部分事件关联回外发消息，但没有文档化的 `conversationId`、`billingConversationId` 或 “single message 转 conversation”的专用字段。因此，系统要自己维护消息事件表和 24 小时会话窗口，再把本地归类结果与供应商交易接口做核对。

**常见设计取舍**：只存一张 webhook 明细表，开发最省事，但后续既难算账，也难解释某条消息为什么被算进某个 conversation；拆成 `message_event` 和 `conversation_window` 两层，建模更复杂，但能同时支持回放、审计和近似计费。

**排障经验**：遇到问题：想直接从 RCS webhook 判断“是哪条 single message 转成了 conversation”。原因：默认认为 webhook 会返回显式会话主键。解决方案：先承认文档没有提供这个字段，强制所有外发写唯一 `reference`，入站事件尽量保存 `messageContext`，再用本地 24 小时窗口和供应商账单数据做补充判断。

**可复用经验**：凡是第三方平台把“事件通知”和“计费归类”分开放文档，就不要用 webhook 当权威账单源。正确做法是：webhook 负责捕获事件，账单接口负责对账，二者之间由本地归并模型衔接。

**延伸思考**：后续可以继续验证 CM.com 是否存在其他交易或结算接口能提供更强的 conversation 级账单主键，降低本地推断成本。

## 主题：给 Agent 暴露数据库能力时，最小权限与 SQL 守卫必须双层收口
**一句话结论**：数据库账号的只读权限负责兜住最终边界，应用层 SQL 守卫负责尽早拦截误用；两层一起存在，数据库工具才适合安全地交给 Agent 或 MCP 使用。

**解决什么问题**：这能降低 Agent 执行数据库查询时误写数据、误发多语句或一次取回过多结果的风险，也让工具在本地调试和上线接入时更可控。

**核心原理**：数据库权限是最后一道硬边界，应用层守卫则负责对输入 SQL 做轻量但高频的约束，例如只允许 `SELECT / WITH / EXPLAIN`、拒绝多语句、收敛返回行数。为了让这套策略能稳定验证，SQL 守卫、limit 处理和配置解析应从 MCP server 入口中拆出来，做成不依赖数据库的纯逻辑模块，先通过单测，再接真实连接层。

**常见设计取舍**：只依赖数据库只读账号，工程上最省事，但错误反馈会更晚，且难以对多语句和返回量做统一约束；再加一层应用守卫，虽然多了实现成本，却能把误用更早挡在工具入口。把所有逻辑都塞进 server 入口文件，写起来快，但测试会被数据库环境绑死；拆出守卫和配置模块，结构更清晰，也更利于后续扩展 AST 校验、查询超时和审计能力。

**排障经验**：遇到问题：数据库工具刚开始设计时，SQL 校验、配置加载和 MCP 启动逻辑混在一起，导致没有真实 MySQL 环境时很难验证行为。原因：把“策略逻辑”和“外部依赖”耦合在了同一层。解决方案：先把 `prepare_readonly_query` 和 `load_config` 抽成纯逻辑模块，用单测覆盖允许语句、拒绝多语句、limit 收敛和配置优先级，再把连接层保持为薄封装。

**可复用经验**：凡是要给 Agent 暴露数据库、搜索引擎、对象存储等高风险读写能力，都适合先做三层拆分：最小权限、应用守卫、薄封装接入层。这样既能提早暴露风险，也能把测试成本控制在没有真实基础设施的范围内。

**延伸思考**：后续可以继续补强 SQL AST 级校验、查询超时、审计日志和慢查询限制，把只读工具从“能用”提升到“可长期治理”。

## 主题：多 Agent 应用应该从“单 Agent + 工具”起步，再按复杂度拆角色
**一句话结论**：多 Agent 的价值来自任务分工和结果复核，不来自角色数量本身；如果任务还没有复杂到需要明确协作，先做 `单 Agent + 工具` 往往更稳。

**解决什么问题**：这能避免项目在一开始就引入过多角色、调度和状态复杂度，导致系统尚未验证业务价值，工程成本先失控。

**核心原理**：单 Agent、固定工作流、多 Agent 解决的是不同级别的复杂度问题。单 Agent 擅长处理中低复杂度任务；固定工作流适合步骤稳定、规则明确的流程；多 Agent 则适合需要拆解、并行、复核和跨能力协作的长链路任务。真正的设计关键是先判断任务能否被稳定拆分，再决定是否引入编排和多角色协同。

**常见设计取舍**：一开始就上完整的 Planner、Router、Reviewer、Executor 能获得更强的可扩展性，但调试、观测和状态管理成本也更高。先以 `Coordinator + 工具` 验证闭环，再把检索、执行、审核逐步拆出去，牺牲的是部分早期结构完备性，换来更低的落地风险。

**排障经验**：遇到问题：项目还在验证需求时，就试图把角色、编排和记忆一次性设计齐全。原因：把“技术上能拆”误当成“当前必须拆”。解决方案：先把用户目标、工具能力和状态管理跑通，确认确实存在多技能协作和独立复核需求，再扩展为多 Agent。

**可复用经验**：做 AI 应用架构选型时，先回答三个问题：任务是否需要拆分、子任务是否需要不同能力、结果是否需要独立复核。三个问题里前两个都不成立时，通常不需要多 Agent。

**延伸思考**：可以继续沉淀一套“什么时候从单 Agent 升级到多 Agent”的判定清单，把业务复杂度、失败代价和治理要求一起纳入决策。

## 主题：Memory / Session 和 RAG 是两条不同的能力链路，边界必须先划清
**一句话结论**：`Memory / Session` 负责管理用户与任务上下文，`RAG / 向量检索` 负责供给外部事实与知识证据；两者如果混成一个系统，后续权限、生命周期和评测都会失真。

**解决什么问题**：这能避免把聊天历史当知识库，也能避免把企业制度、文档和事实资料塞进 memory 体系，导致召回、审计和删除策略都变得混乱。

**核心原理**：Memory 以“当前用户、当前会话、当前任务”为中心，强调短期状态、长期偏好、执行状态和上下文压缩；RAG 以“企业知识与事实资料”为中心，强调数据接入、切块索引、检索、重排、上下文拼装和引用。两者的读取时机、存储方式、权限模型和评测指标都不一样，应该在编排层协同，而不是在存储层混合。

**常见设计取舍**：把所有上下文都塞进一套长期存储，开发上最省事，但后续很难解释“为什么这条回答用了这些信息”。将 Memory、Execution State、RAG 分层设计，工程实现更复杂，却更利于治理、追踪和成本控制。

**排障经验**：遇到问题：系统里“用户偏好”“任务步骤”“制度文档”都被叫作记忆，导致注入上下文时无法判断哪些该长期保留、哪些该按问题实时检索。原因：没有先按信息性质分层。解决方案：先按“用户/任务上下文”“执行状态”“企业事实知识”拆成三类，再分别设计存储、生命周期和读取策略。

**可复用经验**：任何 AI 系统在设计上下文能力前，都可以先做一张边界表：哪些信息需要跨会话记住、哪些只属于当前任务状态、哪些本质上是外部知识检索对象。表一旦清楚，后续的技术选型会顺很多。

**延伸思考**：可以进一步补充 Memory 写入判定、RAG 评测指标和权限继承模型，形成更完整的共享能力层设计规范。

## 主题：长提示词自动化任务必须把执行顺序和检查机制显式写进约束
**一句话结论**：当提示词同时包含执行阶段、输出顺序、资产路径和维护规则时，单靠自然语言提醒不够，必须把“先做什么、做完检查什么、最后核对什么”写成强制流程。

**解决什么问题**：这能降低自动化任务只执行前半段要求、漏掉文件更新路径、漏做知识合并或漏写最终检查的风险。

**核心原理**：模型面对长提示词时，容易优先完成最显眼的输出目标，而弱化后半段的流程性约束。把执行计划、阶段完成检查、最终检查、信息不足处理方式和目标资产路径前置为结构化模板，等于把“注意事项”转化成“必须履行的步骤”。

**常见设计取舍**：自由度高的提示词更短、更自然，但稳定性差，尤其容易在周期任务里出现遗漏。增加强约束会让提示词更长、更像系统规约，却能显著提高输出一致性和资产维护完整度。

**排障经验**：遇到问题：自动化任务输出了摘要，但漏掉了后半段要求的文件维护和最终检查。原因：提示词把流程规则写成了描述性文字，而不是可执行的阶段结构。解决方案：把任务改写成“执行计划 -> 阶段检查 -> 最终检查”的固定骨架，并对每阶段列出输入、输出、更新文件和遗漏风险。

**可复用经验**：凡是要求“既要生成结果、又要维护多个资产、还要遵守输出顺序”的自动化任务，都适合先固化成阶段式模板，而不是依赖模型自由组织。

**延伸思考**：可以继续沉淀一套“周期型知识管理自动化提示词模板”，把范围过滤、资产更新和风险检查做成可复用框架。

## 主题：任务实例化与模板配置是两条不同的生命周期，自动生成任务不应隐式承担配置补建责任
**一句话结论**：只负责生成 `task_list` 的定时任务链路，不应被当成 `template_config` 的创建入口；如果发送阶段按模板配置读取数据，那么模板配置必须在更早的配置阶段或外部同步阶段就已就绪。

**解决什么问题**：这能避免把“发送链路读不到配置”误判成“任务生成没写配置”，尤其适合区分任务实例化、配置管理、缓存刷新和外部同步之间的职责边界。

**核心原理**：任务生成链路的输入通常是已生效的任务配置和触发时间片，输出是任务实例；模板配置则属于发送来源和运营配置资产。若发送前按 `taskConfigId + language` 从缓存或配置表读取 `template_config`，那么 `template_config` 就是发送前置依赖，而不是任务生成副产品。仓库里如果只有 query/cache refresh 而没有 insert/save path，通常说明当前 bounded context 是“消费配置”而不是“生产配置”。

**常见设计取舍**：把 `template_config` 在生成任务时顺手补齐，看似减少依赖，实际会把配置生命周期和任务生命周期耦合，导致历史配置溯源、配置修改回放、幂等和缓存一致性更复杂。把 `template_config` 明确保留在配置阶段或外部同步阶段，职责更清晰，但排查时必须接受“写路径可能不在当前仓库”这一现实。

**排障经验**：遇到问题：顺着任务自动创建方法往下看，只发现 `task_list` 写入，却一直找不到 `template_config` 的新增逻辑。原因：默认把“发送会用到的配置”误认为“应该在任务生成时创建”。解决方案：反过来根据读取链判断 source of truth，确认发送前真实读取的是 `template_config` / cached config，再把排查重点转到配置阶段、外部同步或其他 bounded context，而不是继续在生成链路里找不存在的写入。

**可复用经验**：先画两张图，一张是“谁创建任务实例”，一张是“发送时真正读取谁”。两张图的对象不一致时，优先承认这是多生命周期模型，再去找各自的创建入口。排查缺配置问题时，先区分“写路径不在当前仓库”和“当前仓库写漏了”。

**延伸思考**：如果业务要求“任务生成后冻结当时模板配置”，更适合新增显式快照层，而不是让 `template_config` 的配置生命周期偷偷混入任务生成链路。

**关联知识**：上游是“配置生命周期建模”和“缓存一致性边界”；下游是“任务快照设计”和“发送前读取链分析”；替代方案是“生成任务时做显式配置快照”；项目实践是 `voghion-marketing-push` 的 `SmsTaskList` / `SmsTaskTemplateConfig` 链路；最佳实践是先按读取源判断真实归属，再找写路径。

## 主题：字段命名如果依赖 ORM 默认映射，就必须优先服从数据库契约，而不是局部业务口径
**核心概念**：在 MyBatis-Plus 这类依赖“驼峰转下划线”默认规则的项目里，Java 字段名本身就是数据库列名的一部分。如果数据库真实列是 `credential_id`，模型层却写成 `channelCredentialId`，系统就会自然映射到 `channel_credential_id`，哪怕业务上大家说的是同一个概念，持久化语义也已经偏掉了。

**一句话关键结论**：只要项目主要依赖 ORM 默认命名映射，字段命名首先要和真实表结构对齐；否则“名字看起来更语义化”反而会制造持久化错位。

**解决什么问题**：这类判断能避免在模型、数据库、消息对象之间同时出现多个近义字段名，减少“代码能编译，但实际落错列、查错列”的隐性问题，也能降低后续做跨系统透传时的理解成本。

**核心原理**：默认映射框架不会理解“这是同义词”，它只做机械转换。也就是说，命名一旦偏离数据库约定，错误不是在运行时才发生，而是在设计时就已经埋下了。字段名越靠近底层存储契约，越适合直接使用数据库口径；如果确实要保留不同业务语义，就应该显式加映射注解，而不是依赖默认规则去“猜”。

**常见误区/取舍**：常见误区是为了和某个新概念保持一致，直接改 Java 字段名，却忽略了表结构和其他对象还沿用旧契约。取舍上，统一改名的改动面更大，但长期语义更干净；保留旧名再加显式映射，改动较小，但会把“字段名和真实列名不一致”的认知负担长期保留下来。

**可复用经验**：遇到“字段到底该不该改名”的问题时，先检查三件事：数据库真实列名是什么、ORM 是否依赖默认命名规则、跨模块消息对象是否已经存在统一名称。三者如果已经有两个一致，就优先让第三个也对齐。

**后续可继续深化的问题**：可以继续整理一套“字段命名变更决策规则”，明确什么情况下应该统一改名，什么情况下更适合保留旧名并显式做映射。

## 主题：凭证应该固化在“任务快照”还是“模板配置”上，本质取决于发送时按什么粒度做选择
**核心概念**：`credentialId` 放在 `task_list`、`task_config` 还是 `task_template_config`，不是单纯的表结构问题，而是发送流程按什么粒度选配置的问题。如果发送前会先按语言、模板或渠道分支挑选一份细粒度配置，再从该配置组装消息，那么真正决定发送行为的凭证就更适合挂在这层细粒度配置上。

**一句话关键结论**：凭证该落在哪一层，不看“哪张表离任务更近”，而看“发送入口最终读取的是哪一层对象”。

**解决什么问题**：这能避免把凭证提前固化到一个过粗的层级，结果发送时又要按语言或模板重新分流，最终一个字段表达不了真实差异，或者任务生成后看似有快照，发送时却根本不读它。

**核心原理**：如果一个任务可能根据语言命中不同模板配置，而不同模板配置又允许绑定不同账号或不同凭证，那么 `task_list` 上单一的 `credentialId` 无法完整表达运行时选择。反过来，如果业务已经明确一个任务在任何语言分支下都必须共用同一个凭证，那么把凭证固化到任务快照层才真正有意义。设计关键不是“要不要快照”，而是“快照粒度是否覆盖真实分歧”。

**常见误区/取舍**：误区是看到任务最终要发送，就直觉认为所有发送相关字段都应放进任务表。这个判断只有在发送逻辑不再依赖更细粒度配置时才成立。取舍上，把凭证保留在模板配置层更贴近真实发送来源，但历史任务会受后续模板修改影响；把凭证复制到任务层更利于冻结历史状态，但需要确认不会丢失语言或模板级差异。

**可复用经验**：设计“配置 -> 任务 -> 发送”三层模型时，可以先画一张读取链：发送器真正读取谁，缓存里存的是谁，按什么键分流。谁是最终读取源，谁就最值得承载真实生效的凭证字段。

**后续可继续深化的问题**：可以继续对比“模板层保留真实来源 + 任务层附加快照字段”和“完全转为任务层快照”两种方案，在历史回放、一致性、排障便利性上的差异。

## 主题：发送链路里“凭证校验通过”并不等于“真实发送参数完整”，必须把两类校验分层
**核心概念**：很多第三方渠道存在两类不同语义的参数：一类是账号级凭证，适合通过“查询模板、查询账户信息”之类的校验接口提前验证；另一类是发送时才会进入请求体或 header 的运行时参数，只靠凭证校验接口并不能完全证明它正确。

**一句话关键结论**：校验接口能证明“这套账号凭证可用”，不代表能证明“这次真实发送一定可用”；如果运行时参数参与真实请求，就应该单独建模为发送配置或探活校验问题。

**解决什么问题**：这能避免把所有发送失败都误判成“凭证无效”，也能避免因为某个发送期参数无法在凭证校验接口中验证，就强行把它塞进 credential 状态模型，导致语义混乱。

**核心原理**：凭证校验接口只覆盖它能触达的供应商能力边界。若真实发送请求还依赖额外的 `appKey`、模板、业务号码或应用标识，而这些字段只有在真实发送或近似发送时才会被消费，那么系统就必须承认：存在“credential valid，但 send profile invalid”的状态组合。把两者拆开，系统语义才稳定。

**常见误区/取舍**：误区是坚持所有问题都要折叠进一个“凭证验证成功/失败”状态。这样虽然表面统一，实际上把账号问题和发送配置问题混在了一起。取舍上，分层建模会多出一个状态维度，但能显著提升诊断准确度，也更符合供应商接口的真实边界。

**可复用经验**：凡是发现“校验接口通过，但真实发送仍会因某字段缺失而失败”，就要优先判断这个字段到底属于 credential、runtime profile，还是业务消息本身，而不是先补一堆兼容赋值。

**后续可继续深化的问题**：可以继续整理一套“哪些字段属于账号凭证，哪些字段属于发送配置”的判断标准，并设计统一的预检/探活能力来验证后一类参数。

## 主题：凭证应用契约、发送前本地校验契约、真实请求消费契约，三者必须保持一致
**核心概念**：运行时引入 `credentialId` 或凭证注入机制后，系统里至少会同时存在三套契约：applier 会写哪些字段，本地 `validateParam` 会要求哪些字段，第三方请求实际又会读取哪些字段。只要三者中任意两者不一致，就会出现“明明已经注入成功，却仍在本地失败”或“本地通过了，但发出去还是缺字段”的问题。

**一句话关键结论**：排查发送异常时，不能只看字段有没有被赋值，必须同时证明“校验读的字段”和“真实请求读的字段”跟它是一致的。

**解决什么问题**：这能系统性识别两类常见故障：一类是旧校验还要求老字段，而新凭证只补新字段；另一类是前面注入过的值在发送前又被硬编码或重赋值覆盖。

**核心原理**：applier 负责“把新凭证翻译到消息对象”，校验器负责“决定哪些字段非空才允许继续”，发送器负责“真正把哪些字段带到第三方”。这三层任何一层都可能落后于另外两层。只看某一个环节是否正常，会高估链路一致性；只有把写入、校验、消费三个视角同时放在一张表里，才能看清真实契约。

**常见误区/取舍**：误区是把问题归因为“某个 applier 少赋了两个字段”，但实际上真实问题可能是校验逻辑还停留在旧模型；反过来，也可能不是校验老，而是发送器后续写死了默认值，把注入结果覆盖掉了。取舍上，做这种三层对齐审查前期更费时间，但能避免反复修补局部兼容逻辑。

**可复用经验**：今后遇到字段链路问题，可以固定按三列审查：`applier 写了什么`、`validate 读了什么`、`send 真正用了什么`。只有三列一致，才能说明契约闭合。

**后续可继续深化的问题**：可以继续沉淀一套统一的发送前契约检查机制，把渠道差异包装起来，避免每个供应商各自维护一套容易漂移的本地校验。

## 主题：新增父记录后立刻创建子记录时，先把生成的主键回写到内存对象，再组装子对象
**核心概念**：很多“子表关联字段默认为 0”或为空的问题，并不是数据库没生成主键，而是应用在主记录批量保存后，没有把回填出来的主键同步回原始 DTO 或中间对象，就立刻拿旧对象去创建子记录了。

**一句话关键结论**：只要子对象的外键来源于“刚保存出的父对象 ID”，就要先完成父对象 ID 回填，再进入子对象构造阶段。

**解决什么问题**：这能避免新增场景下子记录错误关联到默认值、空值或旧对象状态，也能解释为什么“修改流程没问题，新增流程才出错”。

**核心原理**：修改场景下，父对象原本就带主键，因此后续逻辑即便没有再次同步，也常常表现正常；新增场景下，主键只有在持久化后才出现，若后续继续沿用保存前的 DTO/VO，就会把“还没更新过主键的旧内存状态”带入子记录构造。问题不在数据库，而在对象状态同步时机。

**常见误区/取舍**：误区是把这类问题理解成“需要重查数据库建映射”或“批量保存不可靠”。很多时候更小、更对的修复只是保存后回填一次 ID。取舍上，重查数据库更稳妥但更重，直接同步内存对象更轻量，但前提是父子对象顺序和映射关系清晰可证。

**可复用经验**：只要代码里出现“先 `saveOrUpdateBatch` 父对象，再根据原 DTO 构造子对象”的模式，就值得立即检查：父对象新生成的 ID 是否已经写回 DTO 或中间模型。

**后续可继续深化的问题**：可以继续总结哪些批量保存场景适合直接回填内存对象，哪些场景必须通过唯一键重查数据库做二次映射。

## 主题：把 AI 用在链路迁移和字段审计时，任务必须改写成“可枚举、可证明、可复核”的格式
**核心概念**：AI 很擅长补一处代码，却不天然擅长维护整条链路上的字段不变量。要让它更可靠，关键不是抽象地要求“仔细检查”，而是把任务改写成必须输出渠道清单、字段链路表、问题分类和证据的结构化审计任务。

**一句话关键结论**：对 AI 说“完整梳理一下”远远不够；只有把检查范围、输出格式和证明义务写死，AI 才更接近一个能做链路审计的 reviewer，而不是只会局部修补的实现器。

**解决什么问题**：这能减少 AI 只盯着当前报错点、漏掉其他同型链路的问题，也能让复核从“主观感觉没问题”升级为“列出证据说明为什么没问题”。

**核心原理**：链路迁移的难点不在单个 if-else，而在全局字段约束。AI 默认按局部上下文工作，如果用户不给出强约束，它会自然停在“眼前这个方法已经修好”的层面。通过强制要求它列出消费者入口、重建对象点、凭证应用点、校验点、真实请求点，以及对每个结论给出方法级证据，等于把“全局不变量检查”显式外化成任务结构。

**常见误区/取舍**：误区是把“实现 + 自检”混在一次请求里，希望 AI 顺手也把全局复核做完。更稳的做法通常是分两轮：第一轮只实现，第二轮强制切换到 reviewer 视角做反向审计。取舍上，这会多花一次对话成本，但比反复返工更省。

**可复用经验**：可以固定让 AI 按“重建点 -> 注入点 -> 校验点 -> 请求点 -> 覆盖点”五层去审字段流，并要求它把问题明确分类为“漏传”“覆盖”“契约不一致”三类。这个方法对消息系统、支付链路、第三方 API 封装都适用。

**后续可继续深化的问题**：可以继续沉淀一套团队内通用的“AI 链路审计提示词模板”，并结合不同项目预填常见消息对象名和关键字段名，降低每次手工组织提示词的成本。

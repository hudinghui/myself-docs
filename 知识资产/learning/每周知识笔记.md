# 2026-W24（06-08 ~ 06-14）

## 配置生命周期、任务快照与凭证建模

### 一句话结论
只要系统同时存在“配置选择”和“任务执行”两层语义，就必须明确谁负责选择、谁负责冻结、谁是执行唯一读取源；否则凭证、模板和任务状态会在运行时继续漂移。

### 这个主题本周真正解决了什么问题
解决了营销推送链路中模板配置、任务快照和凭证字段职责混杂的问题，避免把“发送会读取的配置”误判成“应由任务生成链路补建的配置”，也避免把已生成任务再次受后续配置修改影响。

### 关键原理与边界
编辑态配置和执行态快照是两套不同语义。编辑态关注运营如何配置，适合承载模板和凭证来源；执行态关注某次任务到底用什么，适合承载冻结后的执行真相。只有业务已经收敛为“一个执行任务只使用一个凭证”时，任务快照才足以表达真实执行态；如果仍存在语言、模板或渠道级差异，就不能用过粗的快照模型强行覆盖。

### 关键设计取舍
把凭证只留在配置层，改动更小，但历史任务会受后续配置变化影响；把凭证冻结到任务层，更利于重试、审计和补发，但前提是执行阶段真正只认任务快照。模板配置如果同时承担“编辑态来源”和“执行态真相”，短期省字段，长期会让职责和排障边界失真。

### 典型排障或判断方法
先画读取链，确认发送阶段真实读取谁，再追写入链。固定按四层审计迁移是否落地：表结构有没有列、实体类有没有字段、任务生成时有没有写入、执行阶段有没有只从任务表读取。只要任一层脱节，就视为半迁移状态。

### 可直接复用的方法
- 先区分配置资产、任务实例、发送期依赖，再决定字段归属。
- 涉及“配置选择 + 任务执行”的系统，优先定义“谁负责选择、谁负责冻结、谁是唯一读取源”。
- 迁移审计统一用 `schema -> entity -> snapshot write -> runtime read` 闭环检查，而不是只搜字段名。

- 对应技术知识库中的“1.1 任务配置生命周期建模”“1.2 凭证建模与发送链路”“1.3 实体类与表结构契约一致性”。

## 链路迁移、字段审计与对象状态同步

### 一句话结论
迁移问题最常见的失败方式不是方案本身错，而是字段在 schema、实体、复制链路、运行时读取和对象回填之间只改了一半；审计必须证明契约闭合，而不是只证明局部代码改过。

### 这个主题本周真正解决了什么问题
解决了凭证迁移、父子对象保存和多渠道字段收口中“代码看起来支持新方案，但运行时仍读旧值、写空值或关联错位”的问题，把局部修补升级成可证明的链路审计。

### 关键原理与边界
ORM 和 Bean 复制默认信任模型定义，因此“表里有列、实体没字段”或“实体有字段、表里没列”这类问题往往不会第一时间显式报错。新增场景下，父记录主键只有在持久化后才生成；如果没有把新 ID 回写到后续复用对象，子记录就会继续带着旧状态落库。只有把字段和对象状态都视作链路契约的一部分，迁移才算真正完成。

### 关键设计取舍
一轮内同时实现和审计更快，但很容易被当前上下文偏见带偏；拆成“实现”和“复核”两轮更稳。父子关联问题上，直接回填内存对象是最小修复路径，重查数据库建映射更稳但也更重，只有在顺序或映射关系无法可靠证明时才需要。

### 典型排障或判断方法
固定按三类契约排查发送字段问题：`applier` 写了什么、`validate` 读了什么、`send` 真正用了什么。对父子关联问题，优先检查“新增异常、修改正常”是否由主键回填时机导致。对多渠道迁移，统一列渠道清单和字段链路表，再把问题分类为“漏传”“覆盖”“契约不一致”。

### 可直接复用的方法
- 跨渠道迁移先出字段链路表，再改代码。
- 所有“确认无问题”的结论必须附带扫描范围、读取字段和判定依据。
- 只要出现“先保存父对象，再拿旧 DTO 组装子对象”的模式，就先检查主键是否已回写。

### 与现有知识体系的关系
- 对应技术知识库中的“1.3 实体类与表结构契约一致性”“1.4 对象状态回填与父子记录关联”“2.2 链路迁移与字段审计”。
- 与工程方法论中的“字段、校验和发送行为不一致，导致链路看似赋值成功却仍发送失败”“新增父记录后立刻组装子记录，子表关联字段落成默认值”“AI 参与迁移改造时容易修单点、漏同型链路”直接相关。

## AI 工程化协作与数据库能力开放

### 一句话结论
AI 工程化落地的关键不是先把组件堆全，而是先把边界画清：数据库能力要靠最小权限加输入守卫双层收口，架构选型要从 `单 Agent + 工具` 起步，Memory、Execution State 和 RAG 必须分层。

### 这个主题本周真正解决了什么问题
解决了 Agent 能力开放、架构演进和上下文体系设计中“安全边界、状态边界和角色边界混在一起”的问题，让工具、状态和知识来源各自有稳定职责。

### 关键原理与边界
数据库账号权限是最终硬边界，应用层 SQL 守卫是入口软边界，二者不能互相替代。多 Agent 不是更高级的默认答案，只适合真实存在角色分工、长链路协作或独立复核的场景。`Memory / Session` 负责用户与任务上下文，`Execution State` 负责任务状态，`RAG / Retrieval` 负责外部事实与知识召回，这三者的生命周期和权限模型都不同。

### 关键设计取舍
只依赖数据库只读账号最省事，但误用反馈更晚；再加应用守卫，工程成本更高，但能更早拦截风险。直接上多 Agent 扩展性更强，但治理成本也更高；先用 `单 Agent + 工具` 验证闭环更稳。把所有上下文塞进统一记忆实现最简单，但后续审计、权限和删除策略都会混乱；分层设计更复杂，但可治理。

### 典型排障或判断方法
数据库工具异常先检查 SQL 守卫、配置优先级和 `stdout` 日志污染。AI 架构问题先问三个问题：任务是否需要拆分、子任务是否需要不同能力、结果是否需要独立复核。上下文污染时，先检查任务状态有没有从自然语言 memory 中独立出来。

### 可直接复用的方法
- 高风险工具统一按“最小权限 -> 应用守卫 -> 薄封装”三层设计。
- AI 架构先从 `单 Agent + 工具` 验证，再决定是否升级到角色拆分和编排治理。
- 先做信息边界表，再做 Memory、Execution State、RAG 的技术选型。

### 与现有知识体系的关系
- 对应技术知识库中的“2.1 MCP 数据访问工具设计”“2.3 多 Agent 应用架构”“2.4 Memory Session 与 RAG 边界”。
- 与工程方法论中的“需要把数据库查询能力开放给 Agent，但又要把误写和误用风险压到最低”“企业级 AI 应用里多 Agent、Memory 和 RAG 容易边界混乱”直接相关。

## 周期型知识自动化与知识资产分层

### 一句话结论
周期型知识自动化必须先建执行状态和门禁，再做正式输出与资产写入；而日、周、月知识资产必须按抽象层分工，不能把不同成熟度的认知混在一层里。

### 这个主题本周真正解决了什么问题
解决了晨间自动化容易重复处理、漏处理、提前写资产，以及知识资产层次混乱的问题，让“恢复工作现场”和“沉淀稳定知识”有了清晰分层。

### 关键原理与边界
只要任务带有“昨日 / 补跑 / 已处理日期”这类周期语义，就必须显式建模 `last_success_date`、待处理窗口、门禁条件和终止规则。日笔记适合保留阶段性认知和知识素材，周笔记适合沉淀跨多天收敛出的完整认知，月笔记只保留跨周验证后仍成立的稳定结论。历史状态不可验证时，只能安全降级，不能补跑未知缺口。

### 关键设计取舍
不维护状态最省事，但断档后无法判断该补哪天；显式维护状态增加复杂度，却能把补跑和终止变成可验证规则。把所有认知直接塞进知识库沉淀更快，但会把未验证内容和稳定结论混在一起；分层维护增加汇总工作，但知识演化路径更清晰。

### 典型排障或判断方法
固定按 `读取 last_success_date -> 计算待处理窗口 -> 判断是否继续执行` 做阶段 0 门禁。对长提示词自动化，先检查是否写成 `执行计划 -> 阶段检查 -> 正式产出 -> 最终检查` 的结构化流程。对周 / 月汇总，优先用“目标标题是否存在”判断是否已汇总，而不是依赖模糊的人工确认。

### 可直接复用的方法
- 周期任务先做状态机设计，再设计输出模板。
- 正式输出和资产写入拆成顺序阶段，所有资产独立判断写入条件。
- 日、周、月知识资产分别承担素材、完整认知、稳定认知三层职责。

### 与现有知识体系的关系
- 对应技术知识库中的“2.5 长提示词自动化任务设计”。
- 与工程方法论中的“复杂自动化提示词容易只执行前半段，后续资产维护和检查被遗漏”直接相关。

## 第三方消息平台计费建模与会话归并

### 一句话结论
第三方平台里的消息方向、事件类型、计费分类和会话归并是四套不同语义；只要 webhook 不提供明确会话主键，就不能把 webhook 直接当成权威账单源。

### 这个主题本周真正解决了什么问题
解决了 RCS webhook 场景下把 `in/out` 误当计费分类、把单条消息误当会话账单，以及在没有 `conversationId` 时试图直接从 webhook 计算费用的问题。

### 关键原理与边界
`in / out` 只说明消息方向，`Basic / Single / Conversation` 才是计费归类；二者正交。`reference` 和 `messageContext` 只能支持部分事件关联，不能天然替代会话级账单主键。没有文档化的 `conversationId` 时，本地只能做基于窗口规则的会话归并，再与供应商交易或账单接口对账。

### 关键设计取舍
只存一张 webhook 明细表开发最快，但后续既难审计也难对账。拆成 `message_event` 和 `conversation_window` 两层，本地设计更复杂，但更适合回放、归因和近似计费。直接把 webhook 当权威账单源省接口接入成本，但会把信息边界问题隐性埋掉。

### 典型排障或判断方法
先分离四个维度：方向、事件、计费、会话归并。出站统一写唯一 `reference`，入站保留 `messageContext`、时间戳和原始 `payload`。对账异常时，先核对本地归并结果，再比对供应商交易或账单接口，不直接从 webhook 推断最终账单。

### 可直接复用的方法
- 第三方平台只要把事件通知和计费归类分开文档化，就不要用 webhook 充当权威账单源。
- 本地建模优先拆成事件事实层和会话归并层。
- 没有会话主键时，先承认信息边界，再引入本地规则和供应商接口补证。

### 与现有知识体系的关系
- 对应技术知识库中的“4.1 CM.com RCS webhook 与计费建模”。
- 与工程方法论中的“第三方 webhook 不返回显式会话主键时，如何做 RCS 会话归并和日账单估算”直接相关。

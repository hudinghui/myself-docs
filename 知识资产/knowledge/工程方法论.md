# 工程方法论

## 问题：需要把数据库查询能力开放给 Agent，但又要把误写和误用风险压到最低
### 分析方法
- 先分清两类边界：数据库账号权限是最终硬边界，应用层 SQL 守卫是入口软边界，二者职责不能互相替代。
- 把工具实现拆成“策略逻辑”“配置解析”“MCP / 数据库接入”三层，判断哪些逻辑必须先在无外部依赖条件下完成验证。
- 明确工具的最小能力面：是否只需要 schema 元数据和只读查询，是否需要多语句支持，是否需要限制返回量。

### 解决路径
- 先用只读数据库账号兜住最终权限边界，再在应用层限制语句前缀、拒绝多语句并收敛 `limit`。
- 将 `prepare_readonly_query`、配置解析等纯逻辑单独实现并配套单测，确认规则稳定后再接入 MySQL 连接和 MCP server。
- 用资源接口先暴露 schema，再开放查询工具；STDIO 场景下统一把日志写到 `stderr`，避免污染协议流。
- 在接入文档中明确配置优先级和运行方式，降低本地联调时的歧义成本。

### 经验总结
- 给 Agent 开数据库能力时，安全设计的核心不是“能不能查”，而是“误用会被挡在哪一层”。
- 把策略层做成可单测纯模块，比一开始就连真实数据库更适合验证工具边界。

## 问题：发送链路里用到了配置，但代码中找不到配置写入路径
### 分析方法
- 先看发送阶段真实读取谁，而不是先看谁“理论上应该写入”。
- 把系统拆成“配置生产链路”“任务实例化链路”“发送消费链路”三条线分别建模。
- 对照仓库中的读写行为，判断当前上下文是配置生产者还是配置消费者。

### 解决路径
- 从读取链反推 source of truth。
- 若当前仓库只消费配置，则转向配置中心、同步任务或其他 bounded context 查找写路径。
- 若确实要求实例冻结配置，则新增显式快照模型，不在原链路上做隐式补建。

### 经验总结
- 缺配置问题，先做生命周期建模，再做代码搜索。
- “找不到写入逻辑”不等于“系统漏写”，也可能是职责划分本来就不在当前服务。

## 问题：字段、校验和发送行为不一致，导致链路看似赋值成功却仍发送失败
### 分析方法
- 固定建立三列表：`applier 写了什么`、`validate 读了什么`、`send 实际用了什么`。
- 区分账号凭证、发送配置、业务消息三类字段，不把所有失败都归因到 credential。
- 判断发送入口最终读取对象与字段快照粒度是否一致。

### 解决路径
- 先统一真实生效字段归属，再决定是否需要兼容赋值。
- 建立统一 pre-check / preflight 入口，对业务必填和凭证兼容分层校验。
- 迁移期采用“新逻辑优先、旧逻辑兜底”，保证功能可平滑切换。

### 经验总结
- 字段问题本质是契约问题，不只是命名问题。
- 先收口契约，再补丁式兼容，能显著减少反复返工。

## 问题：多渠道任务从“配置选凭证”迁移到“任务快照凭证”时，如何避免只改了一半
### 分析方法
- 先确认业务约束是否已经固定为“一个执行任务只使用一个 `credentialId`”；如果不是，就不要用单任务快照强行表达多粒度差异。
- 把每个渠道都拆成同一张四层检查表：真实表结构、实体映射、任务生成写入、执行阶段读取。
- 单独核对模板配置实体是否仍在承载执行态凭证，避免“编辑态来源”和“执行态真相”混在一起。

### 解决路径
- 先明确职责分工：配置层负责选择凭证，任务表负责冻结凭证，执行器只从任务表读取。
- 再按渠道逐个收口：补齐配置实体和任务实体的 `credentialId`，删除模板实体里不该承载的执行态凭证字段，确保生成链路显式写入任务快照。
- 最后反向验证执行路径，确认没有残留“从模板配置回读凭证”的旧逻辑。

### 经验总结
- 迁移方案是否落地，不能靠“字段名改了”来判断，必须证明四层契约闭合。
- “表里有列、实体没字段”或“实体有字段、表里没列”通常比显式报错更危险，因为它们会制造表面完成、实际失效的半迁移状态。

## 问题：Kafka 消费者线程异常退出后，节点长期不恢复
### 分析方法
- 沿时间线串联网络事件、线程状态、心跳行为、组协调器判定和告警时间。
- 同时验证消费线程是否存活、心跳是否仍在、`max.poll.interval.ms` 是否超时、是否启用了静态成员。
- 分辨是“线程死亡无自愈”还是“正常再平衡慢”。

### 解决路径
- 把 `poll()`、`doConsume()`、`commitSync()`、`close()` 纳入一致的异常保护。
- 给裸线程配置异常日志、存活探针和自动重启 watchdog。
- 使用静态成员时，明确线程死亡后的恢复策略，必要时改为关闭重建 Consumer。

### 经验总结
- Kafka 故障不能只盯一条异常日志，要按线程、协议、配置三层看。
- 任何允许 fallback 的新机制，都不应在初始化或局部失败时放大成整体不可用。

## 问题：新增父记录后立刻组装子记录，子表关联字段落成默认值
### 分析方法
- 区分新增场景和修改场景，先判断主键是在保存前已有，还是保存后才生成。
- 对照主记录保存后的对象状态，检查后续组装子对象时是否仍在使用保存前 DTO、VO 或中间对象。
- 如果批量保存后主键已回填到实体，再看这份新状态有没有同步回后续链路真正复用的对象。

### 解决路径
- 主表保存成功后，立即把生成的主键回写到 DTO 及其子对象引用，再继续子表构造或更新。
- 仅当父子映射关系无法由顺序或上下文稳定证明时，再通过唯一键重查主表建立映射。
- 保留轻量兜底校验：若存在子资源但主配置 ID 仍缺失，直接中止后续落库并记录日志。

### 经验总结
- “子表外键是 0 / null”很多时候不是数据库没生成 ID，而是应用没把新 ID 同步回内存对象。
- 先回填对象状态，再继续后续逻辑，通常比“重查数据库建映射”更小、更稳。

## 问题：AI 参与迁移改造时容易修单点、漏同型链路
### 分析方法
- 不按“方法流”抽象任务，而按“字段流”拆解：上游输入、重建对象点、注入点、校验点、真实请求点、覆盖点。
- 先列全渠道清单，再为每个渠道建立同格式字段链路表，避免 AI 只在当前报错点停下。
- 要求每个结论附带方法级证据，并把问题固定分类为“漏传”“覆盖”“契约不一致”。

### 解决路径
- 将任务改写成结构化审计：先清单、后表格、再问题分类，全部完成后才允许给修复建议。
- 把“实现”和“复核”拆成两轮，第二轮强制从 reviewer 视角重新审计，而不是复述自己做过什么。
- 复核时固定扫描五类位置：重建对象点、运行时注入点、发送前校验点、真实请求读取点、后续覆盖点。

### 经验总结
- 提醒 AI “完整梳理”远远不够，必须把范围、输出格式和证明义务写死。
- AI 更适合在可枚举、可证明、可复核的任务结构里工作；没有这些约束时，漏检是常态而不是例外。

## 问题：复杂自动化提示词容易只执行前半段，后续资产维护和检查被遗漏
### 分析方法
- 先区分提示词里的三类要求：结果输出、过程约束、资产维护。
- 检查这些要求是否都被写成显式步骤，而不是散落在长文本中的注意事项。
- 对照目标任务，确认是否存在“多个输出段落 + 多个更新文件 + 额外检查规则”的复合结构；如果存在，就默认需要阶段化执行。
- 再判断任务是否带有“上次处理到哪天”的周期状态；如果有，就把 `last_success_date`、待处理窗口和停止规则视为主流程，而不是附加说明。

### 解决路径
- 把任务重写成固定流程：`执行计划 -> 阶段检查 -> 正式产出 -> 最终检查`。
- 为每个阶段显式列出输入、输出、更新文件、检查点和遗漏风险，避免模型自行决定哪些步骤可省略。
- 对信息不足场景预先约定输出方式，要求明确说明缺口和影响，而不是让模型自由猜测补全。
- 对周期任务先执行门禁：`读取 last_success_date -> 计算待处理窗口 -> 判断是否有有效主题`；门禁未通过时直接终止，不进入正式输出和资产写入。
- 将“正式输出”和“资产更新”拆成两个顺序阶段，所有资产都独立判断写入条件，优先落低风险资产。

### 经验总结
- 长提示词任务要靠结构约束，而不是靠“请仔细阅读”这种软提醒。
- 当一个自动化任务同时要求结果生成和资产维护时，检查机制本身就是主流程的一部分。
- 周期自动化的稳定性，更多取决于状态模型是否清楚，而不只是提示词是否足够长。
- 历史不可验证时宁可降级只处理昨天，也不要为了“补齐”去猜测缺口。

## 问题：企业级 AI 应用里多 Agent、Memory 和 RAG 容易边界混乱
### 分析方法
- 先按问题类型拆层：用户与任务上下文、执行状态、外部知识事实、角色协作与治理。
- 再判断任务复杂度，区分适合单 Agent、固定工作流还是多 Agent 协作。
- 最后把权限、观测、失败恢复和人工兜底作为独立治理层补齐，而不是只盯模型交互本身。

### 解决路径
- 从 `单 Agent + 工具` 起步验证闭环，确认确有拆分需求后再引入 Planner、Reviewer、Research 等角色。
- 将 `Memory / Session`、`Execution State`、`RAG / Retrieval` 分层设计，由编排层统一注入上下文与知识。
- 为企业场景补齐共享状态存储、权限隔离、审计日志、成本监控和失败重试策略。

### 经验总结
- AI 架构设计先解决“边界清不清”，再解决“组件多不多”。
- 很多项目不是少一个 Agent，而是少了对上下文、知识、状态和治理的分层建模。

## 问题：第三方 webhook 不返回显式会话主键时，如何做 RCS 会话归并和日账单估算
### 分析方法
- 先分离三个维度：消息方向（`in/out`）、事件类型（message/status/event）、计费分类（`Basic/Single/Conversation`）。
- 再核对供应商文档里哪些字段是稳定关联键，哪些只是辅助上下文；没有文档化 `conversationId` 时，不默认平台会帮你做会话级主键映射。
- 最后判断哪些结果可以靠本地规则推断，哪些必须依赖供应商交易或账单接口做对账。

### 解决路径
- 出站消息统一写唯一 `reference`，入站事件保留 `messageContext`、原始 payload 和时间戳。
- 本地存储拆两层：`message_event` 负责事实事件流，`conversation_window` 负责按 24 小时规则归并会话。
- 每日账单先基于本地归并做近似统计，再和供应商交易/结算数据做校正，不把 webhook 当唯一账单来源。

### 经验总结
- webhook 更适合做事件采集和业务联动，不适合直接充当权威结算系统。
- 只要平台没有返回明确会话主键，就要把“信息边界”当成设计前提，而不是靠猜测把单条消息和会话一一硬关联。

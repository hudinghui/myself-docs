# 技术知识库

## 营销推送系统

### 任务配置生命周期建模
#### 原理
- `task_config` 表达任务定义与触发规则，`task_list` 表达某次实际生成的任务实例，`task_template_config` 表达发送时使用的模板配置。
- 任务实例化链路负责根据已生效配置生成实例，不负责隐式补建模板配置。
- 当发送阶段按 `taskConfigId + language` 或类似键从缓存/配置表中读取模板配置时，模板配置属于发送前置依赖。

#### 实践
- 排查模板缺失问题时，先确认发送链真实读取源，再定位配置创建、同步或缓存刷新链路。
- 如果业务要求“任务生成时冻结模板”，应新增显式快照层，而不是复用模板配置表承担实例快照职责。

#### 最佳实践
- 生命周期建模优先解耦“配置资产”和“任务实例”。
- 缺配置排查优先判断“写路径不在当前上下文”还是“当前上下文写漏了”。

### 凭证建模与发送链路
#### 原理
- 凭证字段的落点由发送阶段真实读取对象决定，而不是由哪张表更靠近任务决定。
- 如果发送阶段按语言、模板、渠道等维度选择配置，凭证应落在能覆盖该选择粒度的对象上。
- “凭证校验通过”不等于“真实发送参数完整”，账号凭证与发送期参数应分层建模。
- 如果业务约束已经收敛为“一个执行任务只使用一个凭证”，则配置层负责选择来源，任务表负责冻结执行凭证，执行阶段只认任务快照。

#### 实践
- 新凭证方案推荐采用“新凭证优先，legacy 逻辑兜底”的兼容模式，降低迁移期风险。
- 排查字段链路问题时，同时审查 `applier 写什么`、`validate 读什么`、`send 真正消费什么` 三层契约。
- 多渠道迁移落地时，额外核对“表结构是否有列、实体是否有字段、任务生成是否写入、执行阶段是否只从任务表读取”这四层闭环。

#### 最佳实践
- 先画读取链，再决定快照层级和字段归属。
- 凭证状态、发送配置、业务消息三类字段不要混成一个校验模型。
- 不让模板配置同时承担“编辑态来源”和“执行态真相”两种职责。

### 实体类与表结构契约一致性
#### 原理
- ORM 默认映射和 Bean 复制链路都依赖模型定义；只要表结构、实体类、VO 和复制逻辑其中一层没跟上，迁移就会形成“半对齐”状态。
- 这类问题不一定会立刻报错，常见表现反而是“字段看起来改了，运行时仍然读旧值、写空值或没有真正落库”。

#### 实践
- 对跨渠道或跨仓库迁移，使用统一核对表同时检查 schema、PO/VO、任务生成链路、执行读取链路。
- 搜字段名只能发现“有没有改过”，不能证明“链路已经闭合”；必须把真实库表和运行时读取源一起纳入审查。

#### 最佳实践
- 迁移评审要从“字段命名是否统一”升级为“schema -> entity -> snapshot write -> runtime read”闭环验证。
- 只要发现“表里有列但实体没字段”或“实体有字段但表里没列”，就默认当前方案尚未真正落地。

### 对象状态回填与父子记录关联
#### 原理
- 新增场景下，主记录主键通常在持久化后才生成；如果后续仍沿用保存前 DTO 或中间对象构造子记录，子表外键就可能继续使用默认值或旧状态。
- 修改场景通常天然带主键，因此“修改正常、新增异常”往往不是数据库问题，而是内存对象状态同步时机有误。

#### 实践
- 在 `saveOrUpdateBatch` 或等价批量保存成功后，立即把回填出的主键同步回 DTO 及后续要复用的子对象。
- 只有当父子对象顺序或映射关系无法可靠证明时，才退而求其次通过唯一键重查数据库建映射。

#### 最佳实践
- 发现“子表关联字段为 0 / null”时，先检查父记录保存后是否已把新主键回写到内存对象。
- 优先做最小修复：先回填 ID，再继续沿原有逻辑创建或更新子记录。

## AI 工程化协作

### MCP 数据访问工具设计
#### 原理
- 给 Agent 暴露数据库能力时，至少要同时设计三层：数据库最小权限、应用层输入守卫、MCP / 连接层薄封装。
- 只读数据库工具的应用层守卫应优先覆盖语句类型、单语句约束和返回行数上限，例如只允许 `SELECT`、`WITH`、`EXPLAIN`，拒绝多语句，并对 `limit` 做边界收敛。
- SQL 守卫和配置解析适合抽成不依赖 MySQL 的纯逻辑模块，这样可以先做无外部依赖单测，再接数据库连接和 MCP transport。
- 配置读取优先级应显式约定，避免本地文件、显式配置路径和环境变量之间的行为歧义。

#### 实践
- 用 `schema://tables` 之类的资源暴露 `information_schema` 元数据，先让调用方理解表结构，再开放查询工具。
- 用 `query_readonly(sql, limit)` 这类工具统一返回 `columns`、`rows`、`row_count` 和实际生效的 `limit`，减少调用侧对底层驱动行为的猜测。
- 配置解析可以采用“`MYSQL_MCP_CONFIG` 指定文件 > 项目内 `config.local.toml` > 环境变量默认项”的优先级，兼顾本地开发和部署接入。
- STDIO 模式下不要向 `stdout` 打日志，避免污染 MCP 协议流；若要输出错误信息，应写到 `stderr`。

#### 最佳实践
- 数据库账号本身必须是只读，应用层 SQL 守卫只是补充，不是替代。
- 先把策略逻辑做成可单测的纯模块，再让 server 层保持薄封装，能显著降低接入和回归成本。
- 返回行数要有上限，复杂查询还应配合数据库侧超时和资源限制，避免“只读但高负载”的风险。

### 链路迁移与字段审计
#### 原理
- AI 默认按局部上下文优化，很容易完成单点修复，却漏掉其他同型链路的字段不变量。
- “完整梳理流程”如果没有被约束成清单、表格、证据和分类，AI 往往只能证明控制流走通，不能证明字段契约闭合。

#### 实践
- 在改代码前，先让 AI 列出渠道/供应商清单，以及消费者入口、重建对象点、凭证应用点、校验点和真实请求点。
- 强制 AI 对每个渠道输出字段链路表，逐项说明“上游传入什么、中途在哪里重建或覆盖、最终谁在读取”。
- 把问题固定分类为“漏传”“覆盖”“契约不一致”，并要求每个结论给出方法级证据。

#### 最佳实践
- 将“实现”和“复核”拆成两轮：第一轮只改代码，第二轮切换 reviewer 视角反向审计。
- 任何“确认无问题”的结论都必须附带扫描范围、读取字段和判定依据，而不是只给一句主观判断。

### 多 Agent 应用架构
#### 原理
- 单 Agent、固定工作流、多 Agent 分别适配不同级别的任务复杂度，不应混为一种“更高级”的统一方案。
- 多 Agent 的核心价值是分工、调度、复核和状态追踪，而不是把一个问题拆成更多提示词。
- 最小可行的多 Agent 架构通常以 `Coordinator` 为中心，向外连接规划、检索、执行、写作、审核等角色和共享状态存储。

#### 实践
- 先用 `单 Agent + 工具` 验证业务闭环，再根据真实的拆分需求把检索、执行、审核逐步独立成角色。
- 当任务存在跨系统执行、长链路协作或结果复核时，再引入任务编排、状态管理和 reviewer 角色。
- 企业级落地时，要把权限、观测、失败重试和人工兜底放进整体架构，而不是只设计 Agent 层。

#### 最佳实践
- 先判断任务是否真的需要拆分，再决定是否引入多 Agent。
- 架构升级顺序优先是：`单 Agent + 工具 -> 角色拆分 -> 编排治理`。

### Memory Session 与 RAG 边界
#### 原理
- `Memory / Session` 负责用户偏好、会话上下文、任务状态与长期画像；`RAG / 向量检索` 负责文档、制度、知识库等事实性内容的召回与引用。
- `Execution State` 应从自然语言 memory 中独立出来，否则任务状态和对话记忆会相互污染。
- 两条链路的生命周期、权限模型、读取时机和评测指标不同，不能用一套“统一记忆”概念覆盖。

#### 实践
- Memory 设计优先分层：`Session Memory`、`Short-term Memory`、`Long-term Memory`、`Execution State`。
- RAG 设计优先分层：数据接入、数据处理、索引、检索、生成，并在检索前加入权限过滤和查询改写。
- 编排层统一负责 memory 注入、retrieval 注入、prompt 组装、引用追踪和成本控制。

#### 最佳实践
- 不把聊天历史当知识库，也不把企业知识库存成用户记忆。
- 先画信息边界表，再做 Redis、数据库、向量库、搜索引擎等技术选型。

### 长提示词自动化任务设计
#### 原理
- 长提示词中的流程约束如果只是描述性文字，模型很容易优先完成显眼输出，而忽略后半段规则和资产维护动作。
- 把执行计划、阶段检查、最终检查和信息不足处理写成固定骨架，能显著提高周期任务的一致性和完整性。

#### 实践
- 对复杂自动化任务，先定义阶段编号、目标、输入、输出、更新文件、检查点和遗漏风险。
- 阶段完成后显式核对是否漏字段、漏文件、漏合并规则和信息不足声明，再进入下一阶段。
- 对“既要输出摘要、又要维护多个知识资产”的任务，单独定义最终检查清单，防止遗漏路径或输出顺序。

#### 最佳实践
- 流程性强的自动化任务，优先使用结构化模板而不是自由文本提示。
- 输出要求越多，越要把执行顺序和检查责任前置为显式约束。

## 消息中间件

### Kafka 消费者故障恢复
#### 原理
- 消费线程退出后，Kafka 心跳线程仍可能短时维持成员在线状态。
- 超过 `max.poll.interval.ms` 后，心跳线程会触发离组；若启用了静态成员，Broker 会等待相同 `instanceId` 重连。
- `commitSync()` 等提交操作若未纳入异常保护，网络抖动可直接导致消费线程静默死亡。

#### 实践
- 对消费者建立线程存活监控、异常日志和自动重启机制，避免线程死后无感知。
- 区分“网络抖动导致短时断连”和“线程死亡导致长时间不自愈”两类场景。
- 使用静态成员时，要同步设计 watchdog 或降级策略，避免分区长期悬空。

#### 最佳实践
- 故障分析固定检查：消费线程、心跳线程、Coordinator、静态成员、Lag 与告警时间线。
- 提交、消费、关闭等关键路径统一纳入异常保护和可观测性设计。

## 第三方消息平台集成

### CM.com RCS webhook 与计费建模
#### 原理
- `in / out` 是消息方向，`Basic / Single / Conversation` 是计费分类，二者属于不同维度，不能互相替代。
- CM.com 的入站消息 webhook 有通用结构，但 RCS 文档没有提供显式的 `conversationId`、`billingConversationId` 或“单条消息转会话”的专用标识。
- `reference` 与 `messageContext` 只能支持部分事件关联，不能天然替代会话级账单主键。

#### 实践
- 出站消息强制写唯一 `reference`，入站事件尽量保存 `messageContext`、`channel`、`timeUtc` 和原始 payload。
- 本地建模拆成两层：`message_event` 负责记录每条 `in/out/status/event`，`conversation_window` 负责维护 24 小时会话归并。
- 每日费用统计先基于本地归并结果做近似汇总，再与供应商交易/账单接口做日级对账。

#### 最佳实践
- 不用 webhook 原始记录直接充当权威账单源。
- 方向、事件、计费和会话归并各自分层存储，后续审计和对账才可解释。
- 没有文档化会话主键时，先承认信息边界，再通过本地规则和供应商接口做补充验证。

## 1. 营销推送系统

### 1.1 任务配置生命周期建模

#### 1.1.1 定义与边界
- `task_config` 表达任务定义与触发规则，`task_list` 表达某次实际生成的任务实例，`task_template_config` 表达发送时使用的模板配置。
- 任务实例化链路负责根据已生效配置生成实例，不负责隐式补建模板配置。

#### 1.1.2 核心原理
- 生命周期建模应解耦“配置资产”和“任务实例”。
- 模板配置是否存在，取决于发送链真实读取的前置依赖是否已准备完成。

#### 1.1.3 关键机制
- 当发送阶段按 `taskConfigId + language` 或类似键从缓存或配置表中读取模板配置时，模板配置属于发送前置依赖。
- 如果业务要求“任务生成时冻结模板”，需要单独引入显式快照层。

#### 1.1.4 设计取舍
- 复用模板配置表可以减少数据冗余，但会把“编辑态配置”和“执行态快照”混在一起。
- 引入快照层会增加建模复杂度，但能把实例执行语义固定下来。

#### 1.1.5 排障入口
- 先确认发送链真实读取源，再定位配置创建、同步或缓存刷新链路。
- 缺配置时，优先判断是“写路径不在当前上下文”还是“当前上下文写漏了”。

#### 1.1.6 实践建议
- 在生命周期评审中，先区分配置资产、任务实例和发送期依赖，再决定每层各自承担的职责。
- 需要冻结执行参数时，不要让模板配置表承担实例快照职责。

### 1.2 凭证建模与发送链路

#### 1.2.1 定义与边界
- 凭证字段的落点由发送阶段真实读取对象决定，而不是由哪张表更靠近任务决定。
- “凭证校验通过”不等于“真实发送参数完整”，账号凭证与发送期参数应分层建模。

#### 1.2.2 核心原理
- 如果发送阶段按语言、模板、渠道等维度选择配置，凭证应落在能覆盖该选择粒度的对象上。
- 如果业务约束已经收敛为“一个执行任务只使用一个凭证”，则配置层负责选择来源，任务表负责冻结执行凭证，执行阶段只认任务快照。

#### 1.2.3 关键机制
- 发送链需要区分 `applier` 写什么、`validate` 读什么、`send` 真正消费什么。
- 多渠道迁移时，真实执行参数必须沿着配置选择、任务生成、任务快照、运行时消费这条链闭合。

#### 1.2.4 设计取舍
- “新凭证优先，legacy 逻辑兜底”可以降低迁移期风险，但会提高兼容期的链路复杂度。
- 把凭证状态、发送配置、业务消息混成一个校验模型，会降低定位精度；分层建模成本更高，但语义更清晰。

#### 1.2.5 排障入口
- 字段链路问题先查三层契约：`applier`、`validate`、`send`。
- 迁移问题额外核对四层闭环：表结构是否有列、实体是否有字段、任务生成是否写入、执行阶段是否只从任务表读取。

#### 1.2.6 实践建议
- 先画真实读取链，再决定快照层级和字段归属。
- 不让模板配置同时承担“编辑态来源”和“执行态真相”两种职责。

### 1.3 实体类与表结构契约一致性

#### 1.3.1 定义与边界
- ORM 默认映射和 Bean 复制链路都依赖模型定义；只要表结构、实体类、VO 和复制逻辑其中一层没跟上，迁移就会形成“半对齐”状态。
- 这类问题不一定会立刻报错，常见表现是字段看起来改了，但运行时仍然读旧值、写空值或没有真正落库。

#### 1.3.2 核心原理
- 字段迁移是否真正落地，取决于 schema、实体、快照写入和运行时读取是否同时闭合。
- 只看代码搜索结果，无法证明真实库表与运行时读取源已经对齐。

#### 1.3.3 关键机制
- 模型定义驱动 ORM 映射，复制链路决定字段是否能从上游对象进入下游执行对象。
- 一旦 schema、PO/VO、任务生成链路或执行读取链路其中一层脱节，就会产生“半对齐”行为。

#### 1.3.4 设计取舍
- 只追求字段命名统一，成本低但容易遗漏运行时语义。
- 采用 `schema -> entity -> snapshot write -> runtime read` 闭环验证，成本更高，但能直接验证落地状态。

#### 1.3.5 排障入口
- 对跨渠道或跨仓库迁移，使用统一核对表同时检查 schema、PO/VO、任务生成链路和执行读取链路。
- 只要发现“表里有列但实体没字段”或“实体有字段但表里没列”，就默认方案尚未真正落地。

#### 1.3.6 实践建议
- 字段迁移评审要从“有没有改字段名”升级为“运行时字段链路是否闭合”。
- 把真实库表和运行时读取源一起纳入审查，不要只看代码层。

### 1.4 对象状态回填与父子记录关联

#### 1.4.1 定义与边界
- 新增场景下，主记录主键通常在持久化后才生成；如果后续仍沿用保存前 DTO 或中间对象构造子记录，子表外键就可能继续使用默认值或旧状态。
- “修改正常、新增异常”往往不是数据库问题，而是内存对象状态同步时机有误。

#### 1.4.2 核心原理
- 父子记录关联的关键，不是保存动作是否成功，而是新主键是否及时回写到后续复用对象。
- 修改场景通常天然带主键，因此新增链路更容易暴露状态同步问题。

#### 1.4.3 关键机制
- `saveOrUpdateBatch` 或等价批量保存成功后，持久化层会回填主键。
- 后续子记录若继续使用保存前 DTO 或中间对象，就会把旧状态带入子表外键。

#### 1.4.4 设计取舍
- 直接回填主键到 DTO 和后续子对象，是最小修复路径。
- 当父子对象顺序或映射关系无法可靠证明时，通过唯一键重查数据库建映射更稳，但实现成本更高。

#### 1.4.5 排障入口
- 发现“子表关联字段为 0 / null”时，先检查父记录保存后是否已把新主键回写到内存对象。
- 如果新增异常而修改正常，优先检查状态同步时机，而不是先怀疑数据库保存失败。

#### 1.4.6 实践建议
- 父记录持久化成功后，立即同步回填出的主键，再继续创建或更新子记录。
- 优先做最小修复；只有映射关系无法可靠确认时，再退回数据库重查。

### 1.5 子订单触达场景中的业务锚点与补链查询键

#### 1.5.1 定义与边界
- 在以子订单为中心的退款详情类触达场景里，`showDetailId` 是对外业务锚点，负责承载模板参数、消息 `value` 和退款详情跳转目标。
- `showOrderId` 是内部补链查询键，只用于从子订单补齐主单、地址或其他主单级扩展信息，不应反向覆盖成对外参数。

#### 1.5.2 核心原理
- 业务消息的统一性取决于“用户最终看到和点击的是哪个对象”；对子订单退款详情场景，这个对象就是 `showDetailId`。
- 只要对外业务锚点和内部补链查询键被混用，就会出现模板参数、跳转值和补充查询语义不一致的问题。

#### 1.5.3 关键机制
- reach service 先基于 `showDetailId` 查询子订单，再按需要补链出 `showOrderId` 查询地址或主单扩展。
- `push` 的 `message.value`、`email / iMessage / sms` 的 OneLink `subOrderId` 以及模板参数里的订单字段，应保持与 `showDetailId` 一致。
- 公共业务参数构建可集中处理 `remainingSeconds -> hours`、`expire` 和 `showDetailId`，渠道差异仍由各 sender 负责。

#### 1.5.4 设计取舍
- 统一以 `showDetailId` 做对外锚点，可以保持跨渠道模板和跳转一致；代价是内部若还需主单级数据，必须额外补链。
- 直接把补链后的 `showOrderId` 当成模板参数，短期实现更省事，但会把子订单场景误改成主单语义。

#### 1.5.5 排障入口
- 遇到模板展示单号、落地页 value、订单补链结果不一致时，先检查这三处分别使用的是 `showDetailId` 还是 `showOrderId`。
- 如果 consumer、service、模板配置三处都在“订单号”上有各自定义，先重建“业务锚点 / 补链键 / 收件人补充信息”分类，再逐处核对。

#### 1.5.6 实践建议
- 新增子订单类触达场景时，先显式定义统一业务锚点，再单独设计主单补链逻辑。
- 公共 builder 只放稳定业务语义字段，不把手机号、邮箱、deviceCode、token 这类渠道收件人字段混入其中。

## 2. AI 工程化协作

### 2.1 MCP 数据访问工具设计

#### 2.1.1 定义与边界
- 给 Agent 暴露数据库能力时，至少要同时设计三层：数据库最小权限、应用层输入守卫、MCP / 连接层薄封装。
- 数据库账号本身必须是只读；应用层 SQL 守卫只是补充，不是替代。

#### 2.1.2 核心原理
- 只读数据库工具的输入守卫应优先覆盖语句类型、单语句约束和返回行数上限。
- SQL 守卫和配置解析适合抽成不依赖 MySQL 的纯逻辑模块，先做无外部依赖单测，再接数据库连接和 MCP transport。

#### 2.1.3 关键机制
- 只允许 `SELECT`、`WITH`、`EXPLAIN` 之类的只读语句，拒绝多语句，并对 `limit` 做边界收敛。
- `schema://tables` 这类资源可先暴露 `information_schema` 元数据，再开放查询工具。
- `query_readonly(sql, limit)` 这类接口应统一返回 `columns`、`rows`、`row_count` 和实际生效的 `limit`。
- 配置解析可采用“`MYSQL_MCP_CONFIG` 指定文件 > 项目内 `config.local.toml` > 环境变量默认项”的优先级。

#### 2.1.4 设计取舍
- 把策略逻辑抽成纯模块，会增加拆分成本，但能显著降低接入和回归风险。
- 只依赖数据库只读权限不做应用层守卫，成本更低，但无法覆盖误用和高负载风险。

#### 2.1.5 排障入口
- 工具异常时，先检查 SQL 守卫、配置解析优先级和返回上限是否符合预期。
- STDIO 模式下出现协议异常时，先确认日志是否错误写到了 `stdout`。

#### 2.1.6 实践建议
- 先把策略逻辑做成可单测的纯模块，再让 server 层保持薄封装。
- 返回行数保持上限，复杂查询再配合数据库侧超时和资源限制。

### 2.2 链路迁移与字段审计

#### 2.2.1 定义与边界
- AI 默认按局部上下文优化，很容易完成单点修复，却漏掉其他同型链路的字段不变量。
- “完整梳理流程”如果没有被约束成清单、表格、证据和分类，通常只能证明控制流走通，不能证明字段契约闭合。

#### 2.2.2 核心原理
- 要让 AI 可靠完成迁移审计，必须把审计对象、证据格式和结论分类显式结构化。
- “确认无问题”只有在扫描范围、读取字段和判定依据都清楚时才成立。

#### 2.2.3 关键机制
- 先列出渠道或供应商清单，以及消费者入口、重建对象点、凭证应用点、校验点和真实请求点。
- 再对每个渠道输出字段链路表，逐项说明“上游传入什么、中途在哪里重建或覆盖、最终谁在读取”。
- 将问题固定分类为“漏传”“覆盖”“契约不一致”。

#### 2.2.4 设计取舍
- 一轮完成实现和审计更快，但容易被同一上下文偏见影响。
- 将“实现”和“复核”拆成两轮，成本更高，但更利于反向审计字段闭合性。

#### 2.2.5 排障入口
- 发现字段问题时，不要只看当前修复点，先回溯同型渠道是否共享相同字段链路。
- 审核结论如果没有方法级证据，默认不够可信。

#### 2.2.6 实践建议
- 在改代码前先完成字段链路清单，再进入实现。
- 所有“确认无问题”的结论都附带扫描范围、读取字段和判定依据。

### 2.3 多 Agent 应用架构

#### 2.3.1 定义与边界
- 单 Agent、固定工作流、多 Agent 分别适配不同级别的任务复杂度，不应混为一种“更高级”的统一方案。
- 多 Agent 的核心价值是分工、调度、复核和状态追踪，而不是把一个问题拆成更多提示词。

#### 2.3.2 核心原理
- 多 Agent 架构是否有价值，取决于任务是否真实存在角色分工、长链路协作或结果复核需求。
- 最小可行架构通常以 `Coordinator` 为中心，连接规划、检索、执行、写作、审核等角色和共享状态存储。

#### 2.3.3 关键机制
- 架构升级顺序优先是：`单 Agent + 工具 -> 角色拆分 -> 编排治理`。
- 当任务进入跨系统执行、长链路协作或结果复核阶段，再引入任务编排、状态管理和 reviewer 角色。

#### 2.3.4 设计取舍
- 早期直接上多 Agent，扩展性更强，但系统复杂度和治理成本显著提高。
- 先用 `单 Agent + 工具` 验证业务闭环，虽然能力边界较窄，但更容易验证真实需求。

#### 2.3.5 排障入口
- 先判断任务是否真的需要拆分，再检查是否缺少权限、观测、失败重试或人工兜底。
- 如果多 Agent 方案只是把一个问题拆成更多提示词，通常说明拆分理由不足。

#### 2.3.6 实践建议
- 先验证单 Agent 闭环，再按真实角色边界逐步拆分。
- 企业级落地时，把权限、观测、失败重试和人工兜底放进整体架构，而不是只设计 Agent 层。

### 2.4 Memory Session 与 RAG 边界

#### 2.4.1 定义与边界
- `Memory / Session` 负责用户偏好、会话上下文、任务状态与长期画像；`RAG / 向量检索` 负责文档、制度、知识库等事实性内容的召回与引用。
- `Execution State` 应从自然语言 memory 中独立出来，否则任务状态和对话记忆会相互污染。

#### 2.4.2 核心原理
- Memory 与 RAG 的生命周期、权限模型、读取时机和评测指标不同，不能用一套“统一记忆”概念覆盖。
- 编排层需要统一负责 memory 注入、retrieval 注入、prompt 组装、引用追踪和成本控制。

#### 2.4.3 关键机制
- Memory 可分为 `Session Memory`、`Short-term Memory`、`Long-term Memory`、`Execution State`。
- RAG 可分为数据接入、数据处理、索引、检索、生成，并在检索前加入权限过滤和查询改写。

#### 2.4.4 设计取舍
- 把任务状态混入自然语言记忆实现简单，但会让状态和对话语义相互污染。
- 独立建模 `Execution State` 增加系统组件，但能保持状态边界清晰。

#### 2.4.5 排障入口
- 出现上下文污染时，先检查任务状态是否从自然语言 memory 独立出来。
- 出现检索失真时，优先检查权限过滤、查询改写和 retrieval 注入时机。

#### 2.4.6 实践建议
- 不把聊天历史当知识库，也不把企业知识库存成用户记忆。
- 先画信息边界表，再做 Redis、数据库、向量库、搜索引擎等技术选型。

### 2.5 长提示词自动化任务设计

#### 2.5.1 定义与边界
- 长提示词中的流程约束如果只是描述性文字，模型很容易优先完成显眼输出，而忽略后半段规则和资产维护动作。
- 正式输出和资产写入属于不同阶段：前者负责恢复上下文，后者负责沉淀状态。

#### 2.5.2 核心原理
- 把执行计划、阶段检查、最终检查和信息不足处理写成固定骨架，能显著提高周期任务的一致性和完整性。
- 只要任务带有“昨日 / 补跑 / 已处理日期”这类周期语义，就必须显式建模 `last_success_date`、待处理窗口、终止条件和降级规则。

#### 2.5.3 关键机制
- 先定义阶段编号、目标、输入、输出、更新文件、检查点和遗漏风险，再进入执行。
- 对周期任务，先读取 `last_success_date` 并计算待处理日期窗口，再决定是否查询上下文和继续执行。
- 工作恢复类自动化先以“待处理日期窗口内有更新的线程集合”为外层范围，线程纳入需同时核对工作目录、更新时间和线程摘要，再按主题归类。
- 将资产更新独立判定，并按 `WorkLog -> Daily Notes -> Weekly / Monthly -> Knowledge Base -> Playbook -> Graph` 的顺序串行落盘。
- 周 / 月汇总优先依据“目标标题是否已存在 + 目标文件可读或可创建”判断是否执行，而不是使用不可验证的“是否首次执行”语义。

#### 2.5.4 设计取舍
- 自由文本提示写起来更快，但执行顺序和检查责任容易漂移。
- 结构化模板前期约束更重，但能降低遗漏路径、遗漏字段和重复写入风险。

#### 2.5.5 排障入口
- 周期任务异常时，先检查 `last_success_date`、待处理窗口和门禁条件是否被前置执行。
- 如果输出完整但资产状态混乱，优先检查是否把正式输出和资产写入混在同一阶段。
- 周 / 月汇总行为异常时，先检查目标笔记文件是否可读取、目标标题是否已存在，以及上游日 / 周素材是否足够。

#### 2.5.6 实践建议
- 流程性强的自动化任务优先使用结构化模板，而不是自由文本提示。
- 历史状态不可验证时，只处理 `previous_day`，不要擅自补跑未知缺口。
- 日、周、月知识资产按抽象层分工，避免一层笔记承担多种职责。
- 把“首次执行”这类业务描述转换成文件状态、标题存在性或执行记录这类可验证条件后，再写进自动化规则。

## 3. 消息中间件

### 3.1 Kafka 消费者故障恢复

#### 3.1.1 定义与边界
- 消费线程退出后，Kafka 心跳线程仍可能短时维持成员在线状态。
- 超过 `max.poll.interval.ms` 后，心跳线程会触发离组；若启用了静态成员，Broker 会等待相同 `instanceId` 重连。

#### 3.1.2 核心原理
- `commitSync()` 等提交操作若未纳入异常保护，网络抖动可直接导致消费线程静默死亡。
- 故障恢复需要同时观察消费线程、心跳线程、Coordinator、静态成员和 Lag 的联动状态。

#### 3.1.3 关键机制
- 心跳线程可短时维持成员在线，但超过轮询间隔后会触发离组。
- 启用静态成员时，Broker 会等待相同 `instanceId` 重连，这会改变分区回收和恢复路径。

#### 3.1.4 设计取舍
- 静态成员有助于减少重平衡，但需要同步设计 watchdog 或降级策略，否则分区可能长期悬空。
- 不给提交、消费、关闭路径做统一异常保护，实现简单，但会放大静默死亡风险。

#### 3.1.5 排障入口
- 故障分析固定检查：消费线程、心跳线程、Coordinator、静态成员、Lag 与告警时间线。
- 先区分“网络抖动导致短时断连”和“线程死亡导致长时间不自愈”两类场景。

#### 3.1.6 实践建议
- 为消费者建立线程存活监控、异常日志和自动重启机制。
- 提交、消费、关闭等关键路径统一纳入异常保护和可观测性设计。

## 4. 第三方消息平台集成

### 4.1 CM.com RCS webhook 与计费建模

#### 4.1.1 定义与边界
- `in / out` 是消息方向，`Basic / Single / Conversation` 是计费分类，二者属于不同维度，不能互相替代。
- CM.com 的入站消息 webhook 有通用结构，但 RCS 文档没有提供显式的 `conversationId`、`billingConversationId` 或“单条消息转会话”的专用标识。

#### 4.1.2 核心原理
- `reference` 与 `messageContext` 只能支持部分事件关联，不能天然替代会话级账单主键。
- 没有文档化会话主键时，必须承认信息边界，再通过本地规则和供应商接口做补充验证。

#### 4.1.3 关键机制
- 出站消息强制写唯一 `reference`，入站事件尽量保存 `messageContext`、`channel`、`timeUtc` 和原始 `payload`。
- 本地建模拆成两层：`message_event` 负责记录每条 `in/out/status/event`，`conversation_window` 负责维护 24 小时会话归并。
- 每日费用统计先基于本地归并结果做近似汇总，再与供应商交易或账单接口做日级对账。

#### 4.1.4 设计取舍
- 直接把 webhook 原始记录当权威账单源，接入快，但审计和对账不可解释。
- 将方向、事件、计费和会话归并分层存储，设计更复杂，但后续审计和对账更稳定。

#### 4.1.5 排障入口
- 先区分消息方向问题、事件关联问题、计费分类问题和会话归并问题，不要混成一类异常。
- 对账异常时，先核对本地 `reference`、`messageContext`、窗口归并结果和供应商账单接口。

#### 4.1.6 实践建议
- 不用 webhook 原始记录直接充当权威账单源。
- 方向、事件、计费和会话归并各自分层存储。

#### 4.1.7 直接相关知识

### 4.2 CM.com Transactions API 日账单统计

#### 4.2.1 定义与边界
- 当目标是“按自然日汇总每日应计金额”时，`Transactions API` 比 webhook 更适合作为账单事实源。
- 如果当前业务只需要按天、账号、渠道落汇总金额，并且项目里已有现成的汇总统计记录模型，就不应为了中间国家维度先新增一张持久化明细表。

#### 4.2.2 核心原理
- 交易明细接口表达的是供应商侧已发生的交易，天然更接近账单语义；webhook 表达的是状态、回复、点击等事件，天然更接近业务联动语义。
- 外部分页账单同步只有在“查询窗口固定 + 分页元信息稳定”时才可被视为可信结果，否则应失败重试，而不是继续汇总漂移后的数据。

#### 4.2.3 关键机制
- 以自然日时间窗口分页拉取交易数据，固定查询条件后逐页同步。
- 对 CM RCS 场景，账单统计口径可收敛为 `direction = out`、`status = 20`、`channel = RCS`、`messageType = SINGLE MESSAGE` 的交易集合。
- 第一页初始化 `totalPages` 与总交易数，后续每页都检查这些值是否稳定；若发生变化，立即中断本轮同步。
- 在同步上下文中按 `account -> countryIso -> count` 聚合，再用国家价格表或价格枚举换算金额，最终只输出现有汇总记录需要的结果。

#### 4.2.4 设计取舍
- 直接把 webhook 或回调日志当账单源，接入更快，但国家、计费口径和对账解释性都弱。
- 新增国家级明细表更利于追溯和复算，但会额外引入数据生命周期、补数和一致性成本；若当前只要求日汇总金额，优先复用现有汇总链路更稳。

#### 4.2.5 排障入口
- 先确认账单事实源是否选对：是交易明细接口问题，还是把 webhook 当成了账单源。
- 再检查分页稳定性：`totalPages`、总交易数、自然日查询窗口、页大小是否在本轮同步中保持一致。
- 最后检查过滤口径和国家映射：方向、状态、消息类型、国家码解析和价格映射是否与业务合同一致。

#### 4.2.6 实践建议
- 只要项目里已有按天汇总的账单记录模型，就先让中间国家维度停留在内存聚合上下文，不要为了计算过程先扩表。
- 对第三方分页账单同步增加“分页漂移即失败”的保护，而不是默认所有分页接口都等价于稳定快照。

## 5. Java 服务可观测性

### 5.1 异常链路与日志序列化边界

#### 5.1.1 定义与边界
- `Throwable` 的职责是承载异常链和堆栈信息，不应作为结构化日志 DTO 的一部分做 JSON 深序列化。
- 请求审计日志与异常堆栈日志应分层：前者记录稳定上下文字段，后者记录原始异常链。

#### 5.1.2 核心原理
- MyBatis、连接池和 JDBC 相关异常对象通常会挂住 `MappedStatement`、`DataSource`、`Connection`、`ResultSet` 等复杂内部引用；JSON 序列化器递归访问这些对象时，可能触发驱动内部不支持的方法，从而产生二次异常。
- 一旦“记录异常”本身再抛新异常，表层报错就会偏离原始业务失败点，导致排障先看到的是日志系统故障而不是真正根因。

#### 5.1.3 关键机制
- `@AfterThrowing`、统一异常切面或错误日志 DTO 如果把完整异常对象放进可递归序列化的结构里，再调用 `JSON.toJSONString(...)`，就可能在序列化阶段访问 ORM / JDBC 内部对象。
- 使用 `logger.error("...", e)` 这类原生日志接口时，日志框架只负责打印异常链，不会把异常对象当普通 JavaBean 深度展开。
- 对数据库类异常，如果项目手动装配 `SqlSessionFactory`，还需要显式开启 mapper / `org.apache.ibatis` 日志或指定 MyBatis 日志实现，才能看到 SQL 模板与参数。

#### 5.1.4 设计取舍
- 把完整异常对象直接塞进结构化日志，输出看似统一，但会把日志系统和异常内部引用链耦合起来，放大二次故障风险。
- 只结构化输出 `exceptionClass`、`message`、`traceId`、`uri`、`bizKey` 等安全字段，再让日志框架打印堆栈，信息更分层，也更适合长期排障。

#### 5.1.5 排障入口
- 如果表层异常与业务路径明显不匹配，先检查异常切面、审计日志和统一错误对象里是否存在“记录异常时再次序列化 `Throwable`”的逻辑。
- 对 Java / Spring 服务，优先搜索 `@AfterThrowing`、`ControllerAdvice`、`JSON.toJSONString(errorInfo)`、`setException(e)` 这类组合，而不是先按表层异常名猜业务根因。
- 如果数据库根因仍不清楚，再检查 logback 是否只开了 root `info`，以及 MyBatis 是否有单独的 SQL 日志配置。

#### 5.1.6 实践建议
- 不把 `Throwable`、`ResultSet`、`Connection`、`MappedStatement`、`HttpServletRequest` 等对象直接挂进会被 JSON 深序列化的日志 DTO。
- 请求审计日志只保留稳定字段，异常链统一通过 `logger.error("...", e)` 输出。
- 需要复现数据库根因时，先恢复 SQL 可观测性，再重新跑一次失败请求，不要直接根据被污染的表层异常下结论。

## 关键结论
- 长提示词自动化任务如果没有显式的阶段顺序、检查点和目标资产约束，模型很容易只覆盖前半部分要求。
- 多 Agent 应用不应一开始就追求角色数量，最稳的落地路线通常是先做 `单 Agent + 工具`，再把检索、执行、审核等职责逐步拆开。
- `Memory / Session` 是以用户和任务为中心的上下文系统，`RAG / 向量检索` 是以企业知识为中心的事实供给系统，两者应独立建模并通过编排层协同。
- 给 Agent 暴露数据库查询能力时，数据库只读权限和应用层 SQL 守卫必须同时存在，单独依赖其中一层都不够稳。
- SQL 校验、limit 收敛和配置解析适合拆成独立模块做无数据库单测，MCP wiring 与数据库连接层应保持薄封装。
- 只读数据库工具的本地接入优先级应明确为“显式配置文件路径 > 项目本地配置文件 > 环境变量默认项”，减少集成期歧义。

## 关键结论
- 依赖 ORM 默认映射时，Java 字段命名要优先对齐数据库真实列名，否则会自然映射到错误列。
- 凭证字段该放哪一层，要看发送入口最终读取谁；如果发送按语言或模板分流，任务表单字段不足以表达真实差异。
- 字段链路问题不能只看赋值点，要同时核对 `applier`、`validateParam` 与真实请求三层契约是否闭合。
- 新增父记录后立即创建子记录时，必须先把生成的主键回写到内存对象，再组装子记录，否则容易出现关联字段默认值或空值。
- 让 AI 做迁移改造时，必须把任务改写成“可枚举、可证明、可复核”的审计格式，才能显著降低漏改和漏检概率。
